1Problem reframing — what actually changed

The original tool answered one question: "copy changed files from A to B." The new use case breaks three assumptions at once, and each one changes the architecture.

New reality (from your use case)Assumption it breaksArchitectural consequence
The backup copy is edited while working with clients — it becomes the new source of truth. "Destination is read-only." No longer true — both sides mutate. One-way copy ⇒ bidirectional sync with divergence/conflict detection.
You want to push the backup back into the original folders. "Direction is fixed A→B." Authority can flip; need a baseline to compute a safe three-way merge, plus a mandatory preview/dry-run.
Critical client files need concurrent off-site copies (Google Drive, OneDrive, …). "Single destination." Fan-out replication with per-target queues, retries, and client-side encryption.
Big files, incremental. "Whole-file copy is fine." Content-defined chunking so only changed chunks move — locally and to cloud.
"Very safe, roll back." "Latest copy is enough." Versioned, content-addressed store + journaled, atomic, reversible operations.
The honest risk. Bidirectional sync is a genuinely hard, dangerous class of software (Dropbox/Unison/Syncthing territory). The value of this document is a design where every destructive action is recoverable, conflicts are never auto-resolved by guessing, and the scary parts (reverse-sync, deletions) always go through preview + versioning.

2Design principles (safety is the product)

1 · Version-before-mutate

No file is ever overwritten or deleted until its current bytes are captured in the version store. This single invariant is what makes "roll back" always possible.

2 · Atomic apply

Write to a temp file, fsync, then atomic rename. A crash mid-sync never leaves a half-written client file.

3 · Journaled & reversible

Every operation is written to an append-only journal before it runs. Undo replays the journal backwards. Enables crash recovery too.

4 · Never guess a conflict

When both sides changed the same file, the tool stops and asks. Both versions are always preserved; the "loser" becomes a version, never a deletion.

5 · Preview before reverse/delete

Reverse-sync (backup → origin) and any deletion propagation require an explicit, human-readable plan the operator approves.

6 · Verify by content, not clock

mtime/size is the fast path; a content hash is the source of truth. Clock skew across NAS/cloud never causes a wrong overwrite.

7 · Targets are independent

A slow or failing cloud never blocks the local/NAS copy. Each replica has its own queue and state.

8 · Confidential by default

Client files are encrypted client-side before leaving the machine; credentials live in the OS keychain, never in config files.

3Location roles & topology

The key clarity move: give every location an explicit role. Only two roles participate in the risky two-way merge; clouds are one-way replicas.

Origin editable

The original working folders on the PC. Can change (staff still edit here).

Working copy / SOT editable

Local disk or NAS. The copy used to work with clients. Designated source-of-truth during client work.

Cloud replicas mirror

Google Drive, OneDrive, S3… Redundant, encrypted, off-site. Disaster recovery — not merge participants.

4Component architecture

Layered so the dangerous logic (sync/merge) is isolated from storage and transport, and the transport (cloud) can be swapped without touching sync.

5Change detection for big files

Two-tier detection keeps it cheap for huge files while staying correct:

  1. Fast path — stat. If size and mtime are unchanged vs the index, assume unchanged (current behaviour, kept).
  2. Truth path — content. When stat differs (or on demand / on verify), the file is run through content-defined chunking (FastCDC) and each chunk hashed with BLAKE3. The file's identity is the list of chunk hashes.
Why chunking matters here. A 2 GB client file where one page changed produces a handful of changed chunks. Only those chunks are versioned locally and uploaded to each cloud. This is what makes "incremental" real for big files and for metered cloud bandwidth — and it makes keeping many rollback versions almost free (unchanged chunks are shared).

6Bidirectional sync — the three-way algorithm

Correct two-way sync is impossible with just "newest wins." You need a baseline: the last state both sides agreed on. Comparing each side to the baseline tells you who changed what, and only that lets you detect a true conflict.

  1. Scan Origin (A) and Working (B) → current chunk-hash indexes.
  2. Load Baseline (S) — the committed common state from the last successful sync.
  3. For every path, classify the change on each side vs S: none created modified deleted.
  4. Apply the decision matrix below → a plan of operations (or conflicts).
  5. Version every to-be-overwritten/deleted file, then apply atomically, then verify by hash.
  6. Commit the new baseline; fan out changes to clouds; write the journal.

7Merge strategies (what "suggest merging" means per file type)

Text / code / config

Automatic 3-way merge (diff3) against the baseline. Non-overlapping edits merge silently; overlapping hunks are flagged for manual resolution in the UI.

Big / binary / Office / CAD / media

No safe content merge exists. The tool offers Keep working, Keep origin, or Keep both (auto-rename). The non-chosen side is retained as a version, never lost. "Open both to compare" is one click.

Batch rules

Operator can set defaults, e.g. "for .docx prefer the working copy," or "delete-vs-edit → always keep the edit," to resolve dozens of conflicts at once — with a preview of what each rule will do.

Default posture

When in doubt, keep both. The system's bias is always toward preserving data over cleanliness; cleanup is a later, reversible step.

8Versioning & rollback

Rollback is not a feature bolted on — it falls out of the content-addressed version store (CAS) plus the operation journal.

Per-file history

Every version of a file is a list of chunk hashes + timestamp + origin. Restore any point from a timeline. Cheap because chunks dedupe.

Point-in-time snapshot

Each successful sync writes a labelled checkpoint (a manifest of the whole tree). Restore the entire folder to "yesterday 5pm," Time-Machine style.

Undo last sync

The journal is replayed in reverse: every overwrite/delete is restored from its pre-image version. One-click, atomic.

Trash, not delete

Deletions go to a retention-bounded trash (e.g. keep 30 days), never an immediate hard delete.

Retention policy

Configurable GRFA-style (keep last N, daily for D days, weekly for W weeks). Garbage-collect unreferenced chunks safely.

Verify & scrub

Periodic re-hash of stored chunks detects bit-rot; cloud replicas are checked against expected hashes.

Note on the 2-day audit log. The existing "detailed log, last 2 days" requirement stays for operational audit. It is separate from version retention (weeks/months of file history). Two different clocks, two different purposes — don't conflate them.

9Multi-cloud fan-out

Each cloud is an independent, one-way replica fed from the SOT/CAS. Independence is the safety property: OneDrive being throttled must never stall the NAS copy or block your client work.

  • Per-target queue & state — each provider has its own work queue, cursor, retry/backoff, and health (last-ok, lag, error).
  • Changed-chunks only — replication uploads the same chunk deltas computed once, not whole files.
  • Client-side encryption — critical client files are encrypted before upload (repository-level crypto). The cloud never sees plaintext.
  • Concurrency with backpressure — bounded parallel transfers; large files stream, don't buffer to RAM (keeps the "lightweight" promise).
  • Restore from any replica — a cloud replica is a full restore source if local disk/NAS dies.
Quyết định đã chốt: dùng rclone. File hiển thị/chia sẻ trực tiếp được trên Google Drive/OneDrive (không mã hoá kín repo), để tiện làm việc & gửi khách. rclone là 1 binary Go, tự lo OAuth + resumable upload + backoff rate-limit + 70 provider. Rollback/versioning ta đã tự làm ở Phase 1 (CAS), nên rclone chỉ đóng vai lớp truyền tải.

Cách tích hợp rclone (cụ thể)

  • OAuth desktop — Authorization Code + PKCE, loopback redirect: app mở trình duyệt → user đồng ý → redirect về http://127.0.0.1:<port>/callback → app bắt code → đổi token. rclone có sẵn client_id cho Google/OneDrive nên không cần đăng ký app hay dính "app verification" để bắt đầu.
  • Điều khiển qua daemon — spawn rclone rcd --rc-addr 127.0.0.1:5572 (sidecar ẩn); Rust gọi HTTP: sync/copy, operations/copyfile, job/status, core/stats.
  • Fan-out — mỗi cloud là 1 remote (gdrive:, onedrive:) + 1 hàng đợi riêng trong Rust. Engine tính "file nào đổi" một lần từ metadata DB rồi đẩy vào từng hàng đợi. OneDrive lag không chặn GDrive.
  • Bảo mật token — rclone.conf mã hoá bằng config password; password cất trong OS keychain.
  • Khôi phục — vì file ở dạng thường trên Drive, mất máy thì kéo thẳng từ Drive về cũng được (không phụ thuộc app).
Bẫy Win7 (nếu sau này hạ nền tảng). rclone bản mới (build Go ≥1.21) đã bỏ Win7 → nếu cần chạy Win7 phải ghim bản rclone cũ hơn còn hỗ trợ. Không ảnh hưởng Win10+/mac/Linux.

13Safety & failure handling

Crash mid-sync

Journal + atomic rename ⇒ on restart, incomplete ops are rolled back from pre-images; the tree is always in a consistent state.

File in use / locked

Detect open handles; skip & retry rather than copy a torn file. Report in the plan.

Disk full / quota

Pre-flight space check; transfers are staged, so a full target fails cleanly without corrupting existing data.

Clock skew (NAS/cloud)

Content hash is authoritative; mtime is only a hint. No wrong overwrite from a bad clock.

Accidental mass-delete

Deletion propagation is capped & previewed ("about to delete 4,213 files — confirm?"); everything lands in trash first.

Cloud outage

Replica queue buffers; local work continues; catches up on reconnect. Health dashboard shows lag.

14Phased roadmap

Phase 0 — Harden what existscurrent tool

Keep the one-way incremental backup shipping. Add the version-before-overwrite store + trash. Immediate safety win, no sync risk yet.

Phase 1 — Two-way sync (local/NAS only)the core

Baseline index, three-way diff, conflict detection & review UI, atomic apply, undo-last-sync, per-file history. This delivers "work on the backup, then safely sync back."

Phase 2 — Multi-cloud replicationresilience

Embed Kopia/rclone; Google Drive + OneDrive first; client-side encryption; per-target health; restore-from-cloud.

Phase 3 — Advanced merge & restorepolish

diff3 text merge, batch conflict rules, point-in-time tree restore, retention policies, integrity scrub, scheduling policies per pair.

15Risks & decisions I need from you

Top risk. Bidirectional sync + reverse-writing into the original folders is where data loss lives. The design mitigates it with preview + versioning + journal, but it raises the engineering bar (and testing burden) substantially versus the current one-way tool.
